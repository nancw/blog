    # 数据库

    # 数据库系统DBS

    # 组成

    • 数据库、硬件、软件、人员

    # 体系结构

    • 集中式数据库系统

      • 所有东西都集中在DBMS电脑上

    • 客户端/服务器体系结构

      • 客户端负责请求和数据，服务器负责数据库服务

    • 并行数据库系统

      • 多个物理上在一起的CPU

    • 分布式数据库系统

      • 物理上分布在不同地方的计算机，使用DBMS将所有数据联网管理

      • 结构

        • 分片模式

          • 水平分片

            • 将表中水平的记录分别存放在不同地方

          • 垂直分片

            • 将表中的垂直的列值分别存放在不同地方

        • 全局概念模式

          • 一个整体，相当于概念模式

        • 全局外模式

          • 相当于外模式

      • 特点

        • 分布透明性

          • 用户或应用程序不需要知道逻辑访问的表具体如何分块存储

        • 位置透明性

          • 应用程序不关心数据存储物理位置的改变

        • 逻辑透明性

          • 用户和应用程序无需知道局部使用的是哪种数据模型

        • 复制透明性

          • 用户和应用程序不关心复制的数据从何而来

    # 三级模式-两级映像

    • 三级模式

      • 外模式 --> 视图

        • 对应数据库中的视图，将表进行一定的处理后再提供给用户使用，如用户表中用户名密码组成视图提供给登录模块使用，而用户表中其他列不对该模块开放，增加其安全性

      • 模式 -->基本表

        • 概念模式，通常使用的表这个级别，根据应用、需求将物理数据划分成一张张表

      • 内模式 -->存储文件

        • 管理如何存储物理数据，对数据的存储方式、优化、存放等

    • 两级映像

      • 外模式-模式映像

        • 表和视图之间的映射，存在于概念级和外部级之间，若表中数据发生修改，只需要修改此映射，而无需修改应用程序

      • 模式-内模式映像

        • 表和数据的物理存储之间的映射，存在于概念级和内部级之间，若修改数据存储模式，只需修改此映射，而不需要去修改应用程序

    • 以上为数据库系统是一个分层次设计，从底至上称为物理级数据库（实际为一个数据库文件）、概念级数据库、用户级数据库

    # E-R模式如何转换关系模型

    • 1:1联系，联系可放在任意的两端实体中，作为一个属性（保证1:1两端关联） 1:n联系，联系可以单独作为一个关系模式，也可以再n端加入1端实体的主键 m:n联系，联系必须作为一个单独的关系模式，其主键是m和n端的联合主键

    • 每个实体对应一个关系模式，多少关系模式对应多少张表。注意联系时作为表还是属性，若是属性又是哪张表属性

    • 需要注意

      • 多值属性的处理

        • 某实体具有一或多个属性，则需优化，把该属性提升为实体，通常称为弱实体；或者在转化关系模式时，将实体主键与多值属性单独构成一个关系模式

      • 派生属性的处理

        • 因为派生属性可由其他属性计算，所以转换关系模式时，通常不转换派生属性

      • 在面向对象模型中，关系模式就对应类，关系模式的属性就对应类的属性

    # 关系数据库规范化

    # 函数依赖

    • 定义

      • 给定一个X，能唯一确定一个Y，就称X确定Y，或者说Y依赖于X，例：Y=X*X函数

    • 两种规则

      • 部分函数依赖

        • A可确定C，(A,B)也可确定C，(A,B)中的一部分即(A)可以确定C

      • 传递函数依赖

        • 当A和B不等价时，A可确定B，B可确定C，则A可确定C，是传递函数依赖；
        • 若A和B，则不存在传递，直接就可确定C

    • 价值与用途

      • 非规范化操作会产生问题，数据冗余、更新异常、插入异常、删除异常

    # 键和约束

    • 键

      • 超键

        • 能唯一标识此表的属性的组合

      • 候选键

        • 定义

          • 超键中去掉冗余的属性，剩余的属性就是候选键

        • 求法

          • 根据依赖集画出有向图，从入度为0的节点开始，找出图中一个节点或者一个节点组合，能够遍历完整个图，就是候选键；若不能遍历图中结点，则尝试一些既有入又有出的中间结点并到入度为0的属性集遍历所有结点，集合为候选键

      • 主键

        • 任选一个候选键，即可以作为主键
        • 关系模式中用户正在使用的候选键

      • 外键

        • 其他表中的主键

      • 注意

        • 一个关系的候选键有多个， 但主键只能有一个。 通常在候选键中选一个作为主键

    • 约束

      • 实体完整性约束

        • 即主键约束，主键值不能为空，也不能重复

      • 参照完整性约束

        • 即外键约束，外键必须是其他表中已经存在的主键的值，或者为空

      • 用户自定义完整性约束

        • 自定义表达式约束，如设定年龄属性的值必须在0-150之间

      • 触发器

        • 通过写脚本来规定复杂的约束，本质属于用户自定义完整性约束

    # 范式

    • 数据库中的范式总体概论

    • 第一范式1NF

      • 所有属性都不可以再分割为两个或多个分量

    • 第2范式2NF

      • 当且仅当R是1NF，且每一个非主属性完全依赖主键（不存在部分依赖）时，R就是2NF
      • 典型例子就是候选键是单属性，单属性是不可能存在函数依赖的

    • 第3范式3NF

      • 当且仅当R是2NF，且R中没有非主属性依赖于候选键时，R就是3NF（此时也不存在部分依赖）。一般解决方法为拆分传递依赖的非主属性为一个新的关系模式。
      • 本质是主键要直接决定所有非主属性，不能通过非主属性间接决定。当存在非主属性决定其他属性（与BCFN的区别）的关系

    • BC范式BCNF

      • 当且仅当R是2NF，且R中没有属性依赖传递与候选键时

    • 第4范式4NF

      • 限制关系模式的属性之间不允许有非平凡且非函数依赖的多值依赖，要求把同一表内的多对多关系删除

    • 主属性

      • 候选键内的属性为主属性，其他属性为非主属性

    • 判断为第几范式

      • 首先就是求出该关系模式的主键
      • 其次，判断是否存在部分依赖

    # 模式分解

    • 定义

      • 范式之间的转换一般都是通过拆分属性，即模式分解，将具有部分函数依赖和传递依赖的属性分离出来，来达到一步步优化

    • 分类

      • 保持函数依赖分解

        • 对于关系模式R，有依赖集F，若对R进行分解，分解出来的多个关系模式，保持原来的依赖集不变，则为保持函数依赖的分解。分解后通过传递依赖来实现属性的跨表函数依赖也可以的

      • 无损分解

        • 将关系模式R分解成多个关系模式后，还能还原成原来的关系模式R

        • 方法

          • 公式法

            • 速度快，只能应用一分为2的情况，分解三个关系模式用不上

          • 表格法

            • 速度慢，适合所有形式的模式分解

    # 事务管理

    • 事务提交commit，事务回滚rollback

    • 事务定义

      • 由一系列操作组成，这些操作要么全做，要么全不做

      • 四种特性

        • （操作）原子性

          • 要么全做，要么全不做

        • （数据）一致性

          • 事务发生后数据时一致的，例如，银行转账

        • （执行）隔离性

          • 任一事务的更新操作直到其成功提交的整个过程对其他事务时不可见的，不同事务之间时隔离的，互补干涉

        • （改变）持续性

          • 事务操作结果时持续性的

    # 并发控制

    • 事务时并发控制的前提条件，并发控制就是控制不同的事务并发执行，提高系统效率

    • 存在问题

      • 丢失更新

        • 事务1对数据A进行了修改并写回，事务2也对A进行修改并写回，此时事务2写回的数据会覆盖事务1写回数据，就丢失了事务1对A的更新

      • 不可重复读

        • 事务2读A，而后事务1对数据A进行修改并写回，此时若事务2再读A，发现数据不对，即一个事务重复读A两次，会发现数据A有误

      • 读脏数据

        • 事务1对数据A进行了修改后，事务2读数据A，而后事务1回滚，数据A恢复了原来的值，那么事务2对数据A做的事是无效的，读到脏数据

    # 封锁协议

    • 分类

      • 一级封锁协议

        • 事务在修改数据R之前必须先对其加X锁，直到事务结束才释放

          • 解决丢失更新问题

      • 二级封锁协议

        • 一级封锁协议的基础上加上事务T在读书R之前必须先对其加S锁，读完后即释放S锁

          • 解决丢失更新、读脏数据问题

      • 三级封锁协议

        • 一级封锁协议加上事务T在读数据R之前先对其加S锁，直到事务结束后才释放

          • 解决丢失更新、读脏数据、数据重复读问题

    • 锁

      • X锁

        • 排他锁，若事务T对数据对象A加上X锁，则只允许T读取和修改A，其他事务不能再对A加任何类型的锁，直到释放

      • S锁

        • 共享锁。若事务T对数据对象A加上S锁，则只允许T读取A，但不能修改A。其他事务只能再对A加S锁，直到T释放A上的S锁

      • 两段锁

        • 会造成死锁，死锁可预防，也可解决

      • 注意

        • 解锁的时间不同，会造成预防问题的个数不同，即为不同的封锁协议

    # 数据故障和备份

    # 数据故障

    # 数据备份

    • 静态转储

      • 冷备份，转储期间不允许对数据库进行任何存取、修改操作

      • 优点

        • 非常快速、归档

      • 缺点

        • 只能提供某一时间点上的恢复，不能做其他工作，不能按表或按用户恢复

    • 动态转储

      • 热备份，转储期间允许对数据库进行存取、修改操作，因此，转储和用户事务并发执行

      • 优点

        • 可在表空间或数据库文件级备份，数据仍可使用，达到秒级恢复

      • 缺点

        • 不能出错，否则后果严重，若备份不成功，所有结果几乎全部无效

    • 海量转储

      • 每次转储全部数据

    • 增量转储

      • 每次只转储上次转储后更新过的数据

    • 日志文件

      • 在事务处理过程中，DBMS把事务开始、事务结束以及对数据库的插入、删除、修改的每一次操作写入日志文件。一旦发生故障，DBMS的恢复子系统利用日志文件撤销事务对数据库的改变，回退到事务初始状态
      • 保证数据库可实时恢复

    # 数据仓库与数据挖掘

    # 数据仓库

    • 特殊数据库，一般将很久以前非必要的数据从数据库中提取出来保存到另外一个数据库中

    • 目的

      • 不是为了应用，而是面向主题的，做数据分析，集成不同表，而且是相对稳定，一般不做修改，同时会在特点的时间点，做大量插入，反映历史变化

    • 作用

      • 用来做数据查询、分析、生成报表
      • 使用数据挖掘工具对历史数据进行挖掘，查找数据间关系，发现剩余价值

    # 数据挖掘

    • 方法

      • 决策树、神经网络、遗传算法、关联规则挖掘算法

    • 分类

      • 关联分析

        • 挖掘出隐藏在数据间的相互关系

      • 序列模式分析

        • 侧重点是分析数据间的前后关系（因果关系）

      • 分类分析

        • 为每一个记录赋予一个标记再按标记分类

      • 聚类分析

        • 分类分析法的逆过程

    # 反规范化技术

    # 规范化操作

    • 可防止插入异常、更新、删除异常和数据冗余
    • 一般通过模式分解、将表拆分，来达到目的
    • 表拆分后，不利于查询，每次查询需关联多张表，严重降低查询效率

    # 反规范化操作

    • 主要就是增加冗余，提供查询效率，为规范化逆操作

    • 技术手段

      • 增加派生性冗余列、增加冗余列、重新组表、分割表

    # 语法关键字，不区分大小写

    创建表create table；修改表 alter table；删除表 drop table； 指定外键 primary key(); 指定外键 foreign key()； 索引 index；视图 view； 数据库查询 select...from...where； 分组查询group by，分组时注意select后的列名要适应分组，having为分组查询附加条件； 更名运算 as； 字符串匹配like，%匹配多个字符串，_匹配任意一个字符串； 数据库插入 insert into...values()；  数据库修改 update...set...where； 数据库删除 delete from...where； 排序 order by，默认升序，降序加关键字desc； 授权 grant..权限..on..数据库对象.to..用户..，允许其权限再赋给另一个用户 with grant option； 收回权限 revoke...on...from； 检查where后的谓词条件 with check option； distinct 过滤重复选项，只保留一条记录； intersect 对两个SQL语句的查询结果做与运算，即值同时存在于两语句才被选出； union 出现在两个SQL语句之间，将两个SQL语句查询结果取或运算，即值存在于第一句或第二句都会被选出；

    # 语法原理

    • select之后的为要要查询显示的属性列名
    • from后面时要查询的表名
    • where后面时查询条件，涉及平均数、最大值、求和等运算，必须要分组
    • group by后面时分组的属性列名，分组的条件使用having，后面跟条件
    • 条件判断数字无需打引号，字符串要打单引号
